# Arbiter IR Design

## Problem

Arbiter currently lowers the same `.arb` program multiple times:

- `compiler/compiler.go` walks the CST and emits VM bytecode
- `transpile.go` walks the CST and emits Arishem JSON
- `eval.go` recompiles segment bodies through synthetic rules
- `expert/compile.go` re-reads declarations to synthesize standard rules
- `flags/parse.go` owns a separate CST interpreter for flag declarations
- `arbiterdecl.go` parses arbiter runtime declarations directly from the CST

That split makes every language feature more expensive. A new operator or
declaration shape has to be reimplemented in multiple consumers, and each
consumer subtly drifts. The compiler and transpiler already duplicate most of
the expression lowering logic.

## Investigation Findings

The original IR proposal was directionally right but too ambitious for the
current codebase:

- It assumed the compiler and transpiler were the only important CST consumers.
  They are not.
- It proposed a fully semantic IR immediately: protobuf schema, symbol table,
  type inference, diagnostics, and backend migration in one shot.
- It treated Arishem JSON as if it could faithfully represent every Arbiter
  construct. It cannot. `let` bindings and aggregate expressions already exceed
  the compatibility surface.

The right move is a staged IR:

1. Land one shared in-process IR for rules, consts, features, segments, and
   expressions.
2. Route the bytecode compiler and transpiler through that IR.
3. Migrate the other CST consumers after the declaration and expression shapes
   stabilize.
4. Freeze protobuf only after the in-process shape proves durable.

## Goals

- Eliminate duplicated CST traversal between the bytecode compiler and
  transpiler.
- Give Arbiter one canonical declaration and expression model in Go.
- Preserve current public APIs: `CompileCST`, `CompileParsed`, `Transpile*`.
- Keep the first landing small enough to ship without refactoring the entire
  runtime.

## Non-Goals For Phase 1

- No protobuf schema yet.
- No language-server-grade symbol table yet.
- No full type inference pass yet.
- No migration of `flags`, `expert`, or arbiter runtime declarations yet.
- No attempt to make Arishem JSON lossless for features it does not support.

## Phased Plan

### Phase 1: Shared Go IR

Implemented in this change.

- New `ir/` package with:
  - `Program`
  - declaration nodes for consts, features, segments, rules
  - flat expression array with stable indexes
  - source spans
- `ir.Lower(...)` converts a CST into that IR.
- `compiler.CompileCST(...)` becomes:
  - CST parse result
  - `ir.Lower(...)`
  - `compiler.CompileIR(...)`
- `transpile.go` lowers once to IR and emits Arishem JSON from the IR.

This gives Arbiter one shared lowering path for the two most duplicated
consumers without forcing the rest of the runtime to move immediately.

### Phase 2: Broaden Consumer Coverage

Planned next.

- Rework segment compilation to read `Program.Segments` directly.
- Move expert-rule extraction off the CST and onto IR declarations.
- Move flag parsing onto IR-backed declaration nodes or add explicit flag IR
  declarations if the existing `flags` package needs richer semantics.
- Add validation and diagnostics on the lowered program.

### Phase 3: Freeze External Schema

Only after phase 2.

- Add `ir.proto`
- Generate protobuf bindings
- Add a stable serialization contract for external tools
- Layer symbol tables, diagnostics, and type information onto the stabilized IR

Freezing the wire format before the internal shape settles would lock in the
wrong abstractions.

## Phase 1 IR Shape

### Program

```go
type Program struct {
    Consts   []Const
    Features []Feature
    Segments []Segment
    Rules    []Rule
    Exprs    []Expr
}
```

The program is declaration-oriented and expression nodes live in a flat array.
Declarations reference expressions by index.

### Declarations

```go
type Const struct {
    Name  string
    Span  Span
    Value ExprID
}

type Feature struct {
    Name   string
    Source string
    Span   Span
    Fields []FeatureField
}

type Segment struct {
    Name      string
    Span      Span
    Condition ExprID
}

type Rule struct {
    Name       string
    Span       Span
    Priority   int32
    KillSwitch bool
    Prereqs    []string
    Excludes   []string
    Segment    string
    Lets       []LetBinding
    Condition  ExprID
    HasCondition bool
    Action     Action
    Fallback   *Action
    Rollout    *Rollout
}
```

Phase 1 intentionally stops at the surfaces used by the existing compiler and
transpiler.

### Expressions

The expression model covers the language features already handled by the VM or
transpiler:

- literals: string, number, bool, null, list
- references: variable/member, const, local, secret
- operators: comparison, logic, collection, string, math, null tests, between
- iteration: quantifiers and aggregates

Each expression has a `Span` and a `Kind`. Child relationships are indices into
`Program.Exprs`.

## Lowering Rules

### Includes

`include` resolution stays in `sourceunit.go` before lowering. The IR sees one
merged source unit.

### Const Resolution

The lowering pass collects top-level const names up front so later references
can lower to `ConstRef` nodes regardless of declaration order.

Backends are free to inline consts. Phase 1 keeps const references in the IR
and lets the compiler/transpiler resolve them when emitting.

### Local Scope

Lowering resolves lexical locals for:

- `let` bindings inside `when` blocks
- quantifier iterator variables inside quantifier bodies
- aggregate iterator variables inside aggregate value expressions

Action parameters only get rule-level `let` scope in phase 1. Quantifier locals
that happen to remain available at runtime after a match are not modeled as a
special static scope yet.

### Spans

Every declaration and expression stores both byte offsets and row/column points.
That is enough to support future diagnostics and editor features without
re-lowering the CST.

## Backend Integration

### Bytecode Compiler

`compiler.CompileIR(program *ir.Program)` is the new backend entry point.

It preserves the existing `CompiledRuleset` layout:

- same intern pool
- same instruction stream
- same rule/action/governance tables

`CompileCST` remains as the compatibility wrapper.

### Arishem Transpiler

The transpiler now reads the IR instead of the CST.

Important limitation: Arishem is still a compatibility target, not a canonical
representation of the full language. Some IR expressions do not have a native
Arishem encoding. Phase 1 keeps behavior pragmatic:

- supported constructs emit structured Arishem JSON
- unsupported constructs may fall back to a raw/source-like representation

That is still better than duplicating another CST walker.

### Other Consumers

Unchanged for phase 1:

- `flags/`
- `expert/`
- `eval.go` segment extraction
- `arbiterdecl.go`

Those paths can migrate after the IR declaration surface stabilizes.

## Why This Cut Is Better

This design is deliberately narrower than the original proposal, and that is a
feature.

- It removes the most obvious duplication immediately.
- It avoids pretending the whole runtime can be migrated safely in one patch.
- It keeps the eventual protobuf plan alive without freezing unstable shapes.
- It improves future tooling by carrying source spans and explicit declarations
  now, not later.

The result is a real intermediate representation in production code, not a
paper architecture that requires a second rewrite before it can land.
